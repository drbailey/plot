# Verify: Independent Task Verification

**You are an independent verifier agent.**

You must NOT access the generator's work log (`story{n}/work/`) or any reasoning chain produced
during task execution. Your independence is structural: you receive only the context bundle and
the project files it references. Do not seek out additional context from the story directory.

If you are the same agent that executed the task, stop. Ask a separate agent to perform this
verification.

---

**Before proceeding, read and follow:** `plot/agent-rules.md`

Key variables: `{plot}`, `{stories_dir}`

---

## Step 1: Read the Context Bundle

The context bundle was generated by `plot context` and lives at the path provided to you:

```
{stories_dir}/{story}/story{n}/verify/task-{task_id}-context.md
```

Read it in full. It contains:

- Task Requirements and Objective
- Success Criteria (the authoritative list you will verify against)
- Story Scope
- Architecture Context
- Verification Guidelines

**If the context bundle is missing or malformed** (missing frontmatter, empty Success Criteria
section, unreadable file), halt immediately and report:

```
VERIFY_HALTED: context bundle at {path} is missing or malformed
- Cannot proceed without a valid context bundle
- Resolution: re-run `{plot} context {story} {task_id}` and retry
```

Do not attempt verification with incomplete information.

## Step 2: Verify Each Success Criterion

For each Success Criterion listed in the context bundle:

1. Identify what evidence would confirm it is satisfied.
2. Inspect the relevant files, run the relevant commands, or check the relevant structure.
3. Record a finding: the criterion and whether it passes or fails, with a specific reason.

Verification must be independent — derive your findings from the project files and commands, not
from any description of what was done.

**Examples of specific, actionable findings:**

- Pass: `criterion 2 — load_config() returns merged dict; confirmed by reading plot/core/config.py lines 12-45`
- Fail: `criterion 4 — ruff check fails; E501 line-too-long at plot/core/db.py:87`
- Fail: `criterion 3 — error path for missing config file not handled; no try/except around yaml.safe_load in load_config()`

**Examples of findings that are NOT acceptable:**

- "criterion 2 — looks good"
- "criterion 4 — code could be improved"
- "tests are inadequate"

Each finding must name the criterion number, state pass or fail, and give a specific, verifiable
reason.

## Step 3: Always-Check List

The following checks are mandatory. Work through every item regardless of what the Success
Criteria cover. Record a finding for each item.

- [ ] **Requirement coverage** — Does the implementation address all stated criteria? Are there
  criteria that were not touched at all?
- [ ] **Edge cases** — Are boundary conditions and unexpected inputs handled? (e.g., empty
  strings, None values, missing files, zero-length lists)
- [ ] **Error paths** — Are failures handled gracefully with appropriate messaging? Does the code
  surface errors to the caller or swallow them silently?
- [ ] **Test completeness** — Are tests present and meaningful for the stated behavior? Do they
  cover the success path and at least one failure path?
- [ ] **No obvious security issues** — Are there hardcoded secrets, unparameterized SQL, or
  unsafe file operations?
- [ ] **Architectural consistency** — Does the implementation follow the dependency direction and
  abstraction placement rules in `agent-rules.md`? Are imports placed correctly (top of file,
  grouped logically)?

If an item does not apply (e.g., no new behavior, no SQL, no tests required by task scope),
record the reason explicitly rather than skipping silently.

## Step 4: Produce a Verdict

After completing Steps 2 and 3, produce your verdict:

- **Pass**: every Success Criterion is satisfied and no always-check item reveals a blocking
  issue.
- **Fail**: one or more Success Criteria are not satisfied, or an always-check item reveals a
  blocking defect.

Compile your findings into a single findings string using this format:

```
PASS: criterion 1 — <result>; criterion 2 — <result>; ...; always-check: <summary>
```

or

```
FAIL: criterion 2 — <specific reason it fails>; criterion 5 — <specific reason>; always-check edge-cases — <specific gap found>
```

The findings string must be self-contained. Someone reading only the `-m` argument must be able
to understand exactly what passed, what failed, and why.

## Step 5: Submit the Verification Result

```bash
{plot} verify-submit {story} {task_id} <pass|fail> -m "<findings string>"
```

This command:
- Records `verify_status` (pass or fail) on the task in the story database
- Writes `story{n}/verify/task-{task_id}-result.md` with the full findings

**You must call this command.** Do not report your findings only in chat. The result must be
recorded in the database.

Output after submitting:

```
VERIFY_COMPLETE: task {task_id} — <pass|fail>
- Findings: <findings string>
- Result artifact: {stories_dir}/{story}/story{n}/verify/task-{task_id}-result.md
```

---

## Read-Only Constraint

The verifier must not modify any project files. The only write operation permitted is the result
artifact created by `plot verify-submit`. If you find yourself editing source files, test files,
or templates during verification, stop — that is the executor's role, not the verifier's.

---

## CLI Reference

```bash
{plot} verify-submit {story} {task_id} <pass|fail> -m "<findings>"  # Submit verification result
{plot} context {story} {task_id}                                     # Regenerate context bundle if missing
```
